## 2. 双模式工作流程

### 2.1 方案设计模式（场景→方案）

**触发条件**：客户描述应用场景、询问"怎么做"、"用什么方案"

**工作流程**：

1. **信息采集（强制前置，不得跳过）** — 在输出任何方案之前，必须先向客户逐一确认以下6项关键信息。只要有任意一项未明确，就停下来提问，等待客户回答后再继续。**严禁在信息不完整时假设默认值并直接输出方案。**
   - 相机数量与安装方式（静止/运动）
   - 应用类型（抓取/纠偏/对位）
   - 机构类型（XYR一体/分离轴）
   - 有无旋转需求（是否需要输出角度偏差）
   - 工具是否共轴（工具中心是否=旋转中心）
   - 标定条件（有无标定板、标定板能覆盖几个相机视野）

   > **采集规则**：客户描述中已明确的项可直接确认；未提及的项必须主动提问。提问时将所有缺失项合并为一次提问，避免多轮反复打扰。

2. **场景分类** — 6项信息全部确认后，沿决策树定位到具体场景
   > 详细决策树见 `${SKILL_DIR}/references/scene-decision-tree.md`

3. **方案输出** — 按方案设计输出模板给出完整方案

### 2.2 问题诊断模式（问题→排查）

**触发条件**：客户描述问题现象、询问"为什么不准"、"精度差怎么办"

**工作流程**：

1. **信息采集** — 向客户确认以下关键信息：
   - 问题现象（偏差大/重复性差/某些位置不准/系统性偏差）
   - 误差数据（具体数值，如"X方向偏差0.5mm"）
   - 系统配置（相机/机构/标定方式）
   - 已尝试的解决方法

2. **5Why分析** — 逐层追问根本原因：
   - 表面现象是什么？
   - 直接原因是什么？
   - 根本原因是什么？
   - 涉及哪些子系统？
   - 是否有相互影响？

3. **排查流程** — 按优先级给出排查步骤：
   - 机构及成像排查（静态测试→动态测试）
   - 标定过程排查（参数/数据/结果验证）
   - 示教过程排查
   - 生产过程排查
   > 详细排查流程见 `${SKILL_DIR}/references/error-analysis.md`

> **标定文件辅助分析**：在排查流程第二步（标定过程排查）时，主动询问客户是否有标定文件（平移旋转标定/N点标定模块生成的 .xml 文件），如果有可帮助系统分析标定数据质量。
> - 若客户提供标定文件（文件路径或从对话框复制的 xml 内容），调用 `${SKILL_DIR}/scripts/calib_analyzer.py` 的 `analyze_calib_file()` 或 `analyze_calib_content()` 函数分析
> - 获取 12 项检测结果，标记 pass/fail
> - 将分析结果通过模板 `{{CALIB_REPORT}}` 占位符写入 HTML 诊断报告（在排查路径卡片之后展示独立标定分析报告卡片）
> - 同时在诊断摘要 `{{DIAG_DESC}}` 和 `{{DIAG_TABLE}}` 中标定分析关键发现，`{{CHECKLIST}}` 中用具体数值指引排查

4. **方案输出** — 按问题诊断输出模板给出排查方案

## 5. 特征提取流水线（概要）

> 详细操作见 `${SKILL_DIR}/references/vm-operation-tips.md`

**核心思路**：粗定位定位区域 → 精定位提取特征 → 抓边获取精确坐标

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 图像采集  │───→│ 粗定位   │───→│ 精定位   │───→│ 抓边输出  │
│ (相机拍照) │    │(高精度匹配)│    │(圆/线+修正)│    │(坐标+角度)│
└──────────┘    └──────────┘    └──────────┘    └──────────┘
```

| 阶段 | VM模块 | 目的 | 关键参数 |
|------|--------|------|---------|
| 粗定位 | 高精度匹配/模板匹配 | 定位ROI区域，修正偏移 | 匹配分数阈值≥0.7 |
| 精定位 | 圆查找+位置修正 / 直线查找+位置修正 | 提取亚像素级特征点 | 边缘过渡带<3像素 |
| 抓边 | 边缘检测→交点/圆心计算 | 输出精确坐标和角度 | 重复性≤1像素 |

**粗定位→精定位搭配原则**：
- 粗定位必须覆盖精定位ROI，确保精定位ROI不跑偏
- 位置修正模块连接粗定位和精定位，用粗定位结果修正精定位ROI
- 抓边选择稳定特征（圆孔>直边>角点），避免毛刺和缺陷干扰

## 6. VM关键操作提示（概要）

> 详细操作提示见 `${SKILL_DIR}/references/vm-operation-tips.md`

### 标定模块

| VM模块 | 适用场景 | 关键参数 | 典型值 |
|--------|---------|---------|--------|
| 平移旋转标定 | 九点/十二点标定 | 标定点数、步长、旋转角度 | 步长覆盖视野1/3以上，旋转角度建议15-30度 |
| 映射标定（同侧多相机） | 同侧多相机互映射到主相机坐标系 | 标定板静止不动 | 两相机**同时**拍同一块标定板，标定板不能移动 |
| 映射标定（上下相机对位） | 上下相机贴合对位，标定板需在两相机间传递 | 标定板只平移不旋转 | 通过生产时固定上/下相机拍照位来保证映射关系成立 |

> **多相机标定准则**：所有多相机场景，辅助相机通过 VM **映射标定模块** 统一映射到主相机坐标系；主相机再通过 **平移旋转标定** 建立到机构物理坐标系的转换。传入单点抓取/单点纠偏/单点映射对位模块的特征点，必须是**主相机坐标系下的像素坐标**，模块内部负责转换到物理坐标系运算。
>
> **映射标定两种场景的核心区别**：
> - **同侧多相机映射**：利用多台相机能同时看到同一区域的特性——将标定板放在重叠视野内静止不动，各相机同时拍照采集对应点，标定板绝对不能移动。
> - **上下相机移动映射**（对位场景）：上下相机视野不重叠，标定板需从上相机位**只平移**到下相机位，分别拍照采集；这种映射关系依赖生产时上相机和下相机拍照位与映射标定时完全一致才能成立。

### 生产计算模块

| VM模块 | 适用场景 | 关键参数 | 典型值 |
|--------|---------|---------|--------|
| 单点抓取 | 抓取场景 | 拍照位、基准像素点、**内部加载标定文件** | 拍照位=标定第五点；像素坐标直接输入模块，模块内部完成坐标系转换 |
| 单点纠偏 | 纠偏场景 | 基准物理坐标、基准像素点、**内部加载标定文件** | **创建基准必须在标定第五点（旋转标定坐标）处拍照**，不能在任意位置；像素坐标直接输入模块，模块内部完成坐标系转换 |
| 单点映射对位 | 对位场景 | 映射标定文件、工件坐标系、**内部加载标定文件** | 示教点为映射标定上相机位取标定板位置，即使上下两个图像坐标系重合的点位 |

### 诊断工具

| VM工具 | 用途 | 关键指标 | 判定标准 |
|--------|------|---------|---------|
| 静态测试 | 成像稳定性 | 像素波动 | ≤1像素 |
| 动态平移测试 | 机构平移重复性 | 物理偏差 | ≤0.02mm |
| 动态旋转测试 | 机构旋转重复性 | 角度偏差 | ≤0.1度 |
| 标定验证 | 标定质量 | 重投影误差(标定平移像素误差和旋转像素误差) | ≤0.5像素 |

## 6. 诊断工具箱

### 重投影误差
```
Error = √((x_calc - x_actual)² + (y_calc - y_actual)²)
优秀：< 0.1像素 | 良好：0.1-0.5像素 | 需优化：> 0.5像素
```

### 重复性测试
```
方法：同一位置连续测量30次
指标：重复精度 = ±3σ
判定：重复精度 ≤ 要求精度/3
```

### 坐标系方向验证
```
1. 图像中向右移动 → 机械手X增还是减？
2. 图像中向下移动 → 机械手Y增还是减？
3. 图像顺时针旋转 → 机械手角度增还是减？
```

### 标定验证关键指标

| 指标 | 标准要求 |
|------|---------|
| 平移像素精度 | < 1像素 |
| 旋转像素精度 | < 0.5像素 |
| 物理平移精度 | < 0.02mm |
| 物理旋转精度 | < 0.05度 |
| XY轴夹角 | 89.8°-90.2° |

## 8. 快速参考

### 常用公式
```
坐标变换：[x_robot]   [a  b  c]   [x_img]
          [y_robot] = [d  e  f] × [y_img]
          [   1   ]   [0  0  1]   [  1  ]

角度计算：θ_robot = θ_img + θ_offset

像素精度：像素精度 = 视野大小(mm) / 图像分辨率(像素)
```

### 精度设计原则
- 像素精度 ≤ 要求精度 / 3
- 重复性误差 ≤ 系统精度要求 / 3

### 三大常规场景与多相机归约准则

定位引导所有场景最终归约到以下三种 VM 底层运算模型：

| 场景类型 | VM 核心模块 | 适用条件 |
|---------|-----------|---------|
| **单点抓取** | `单点抓取` | 单/多相机抓取，物料在工作台，引导机构去目标点 |
| **单点纠偏** | `单点纠偏` | 单/多相机纠偏，物料在机构上，引导机构使物料回基准位 |
| **单点映射对位** | `单点映射对位` | 上下相机贴合，目标特征和对象特征均会发生偏移 |

**多相机场景归约方法**（以双上相机拍对角抓取为例）：
1. 标定：辅相机通过**映射标定**映射到主相机坐标系 → 主相机通过**平移旋转标定**映射到机构坐标系
2. 生产：辅相机特征点转换到主相机坐标系 → 两点/多点联立求中点和角度 → 输入**单点抓取/单点纠偏**模块

> 关键约束：传入单点抓取/单点纠偏/单点映射对位的特征点，必须是**主相机坐标系下的像素坐标**。

### 十二点标定触发条件

需要十二点标定（含旋转中心标定）的判断依据，满足以下**任意一条**即需十二点：

- **有旋转需求**（需要输出角度偏差，R轴旋转补偿）
- **工具偏心**（吸嘴/夹爪中心与R轴旋转中心不重合）

只有无旋转需求时，才使用九点标定。

### 常见问题快速索引

| 问题现象 | 可能原因 | 首要检查 | 参考文档 |
|---------|---------|---------|---------|
| 重投影误差大 | 标定点提取不准 | 图像质量和坐标对应 | 标定场景与原理.md |
| 角度越大偏差越大 | 旋转中心标定不准 | 旋转中心拟合精度 | 标定场景与原理.md |
| 系统性偏差 | 坐标系方向错误 | 图像与机械手坐标系对应 | 分离轴场景总结与兼容方法.md |
| 重复性差 | 机构精度/光源不稳定 | 静态测试+动态测试 | 定位项目误差分析及精度提高.md |

## 9. 方案/诊断结果导出（强制）

> **目的**：每次按 §7 模板给出正式回答后，把这次结果落盘到客户指定的文件夹，方便客户保存、回看、归档、复用。

### 9.1 触发条件

满足以下**任意一条**即触发导出：
- 走完了方案设计模式，按 §7.1 模板输出了完整方案
- 走完了问题诊断模式，按 §7.2 模板输出了完整诊断

**不触发**导出的场景：
- 仅信息采集、追问澄清、闲聊
- 用户明确说"不用保存"/"不用导出"
- 输出内容不完整（如只给了部分章节）

### 9.2 导出路径确认（强制前置）

**在写出文件之前，必须先与客户确认输出文件夹**，按以下规则处理：

1. **默认推荐**：当前用户桌面下的 `export/` 文件夹
   - Windows：`C:\Users\<当前用户>\Desktop\export`（用 `os.path.expanduser("~/Desktop/export")` 推导，避免硬编码用户名）
   - 提问示例：
     > 报告默认导出到桌面 `export/` 文件夹（`<完整路径>`），是否使用该路径？如需自定义，请提供文件夹路径。

2. **用户提供路径**：以客户提供的路径为准，支持绝对路径和 `~` 开头的家目录路径

3. **文件夹不存在**：无论默认路径还是客户提供路径，若文件夹不存在，**直接创建**（`os.makedirs(path, exist_ok=True)`），不要二次询问

4. **路径无效或不可写**：如客户提供的路径非法或无写入权限，告知客户具体原因，请其换一个路径

> **跳过确认的情形**：同一会话内客户已确认过路径，后续输出沿用该路径，不再重复询问。

### 9.3 导出文件名

**文件名规则**：`YYYYMMDD-HHmm_模式_关键词.html`（.sol 文件自动同名生成）
- 模式：`方案` 或 `诊断`
- 关键词：从场景或问题中提炼 5-15 字的中文短语，连字符或下划线分隔皆可
- 示例：
  - `20260115-1430_方案_双相机同侧标定板抓取-不共轴旋转.html` → 自动生成同名的 `.sol`
  - `20260115-1612_诊断_十二点标定后角度越大偏差越大.html`

**文件名冲突**：如同分钟内已有同名文件，追加 `_2`、`_3` 区分。

### 9.4 文件内容结构

导出文件为完整 HTML，结构与 §7 输出模板一致，在 HTML 注释中嵌入元信息：

```html
<!-- 元信息
生成时间：YYYY-MM-DD HH:mm
模式：方案设计 / 问题诊断
用户原始问题：{完整复述用户最初的场景描述或问题描述}
关键参数：{清单列出本次确认的所有信息采集项的值}
-->
```

导出内容即本次按 §7.1 或 §7.2 模板生成的完整 HTML 文件，包含所有卡片、流程图、表格。

### 9.5 导出后告知用户

导出完成后，在对话末尾用一句话告知（使用实际写出的绝对路径）：

> 本次方案已导出到 `<完整路径>/{文件名}`，用浏览器打开查看彩色卡片方案，.sol 文件可用 VisionMaster 4.4 打开。

### 9.6 注意事项

1. **导出前必须确认路径**，除非客户已在当前会话内确认过；不要每次问"要不要保存"，只问"输出到哪个文件夹"。仅在客户主动说"不用保存"时跳过。
2. **创建文件夹是默认行为**：若目标文件夹不存在，直接 `os.makedirs(..., exist_ok=True)`，不要再问。
3. **不要把多次回答合并到一个文件**：每次完整输出对应一个独立文件，方便单独引用。
4. **不要导出信息采集中间过程**：只导出最终模板化的完整结果。
5. **自动同步生成 .sol 方案**：`build_solution_html()` 默认 `auto_generate_sol=True`，HTML 写出后自动调用 CLI 生成同名 .sol 文件。CLI 不可用时自动跳过不影响 HTML 写出。
6. **优质样本同步到 samples/**：当用户确认方案正确、或方案覆盖了 `samples/` 中尚无的场景类型时，将该文件以`模式_场景关键词.html`（不含日期）的文件名复制到 `${SKILL_DIR}/samples/`，供后续同类场景参考。
