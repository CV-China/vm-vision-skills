# VisionMaster 方案全方位性能优化

## 概述

本技能是 VM 性能分析的**总调度台**，串联四个专项技能，输出一份「既见树木、又见森林」的综合报告。

四个专项技能各司其职：

| 维度 | 技能名 | 核心能力 | 输入要求 | 特殊说明 |
|------|--------|---------|---------|---------|
| 🛡️ 脚本 | `vm-script-protection` | 代码防护（try-catch/除零/越界/死循环检测）+ 性能问题 | .sol 方案文件 | ⚠️ 如后续有内存/CPU 分析，默认只记录不修改 |
| 📊 内存 | `perfview-analyzer` | C++ 非托管内存分配热点（模块→函数逐层归因）+ 阶段增长对比 | 运行中进程 PID/名 + 测试场景 | 三阶段采集，每阶段先启动采集再提示用户操作 |
| 🔥 CPU | `vtune-analyzer` | CPU 热点函数/模块排名 + 调用栈分析 | 运行中进程 PID/名（需管理员权限） | 优先启动模式；附加模式仅 native 可见 |
| ⏱️ 耗时 | `vm-execution-time-analyzer` | 流程/模块耗时统计 + 波动贡献指数(VCI) + 交互式图表 | ModuleFrame*.log 目录 | 零进程依赖，最先执行 |

**为什么需要组合分析？** 四个维度不是孤立的。比如：
- 脚本中循环内字符串拼接 → 大量临时内存分配 → GC 压力 → CPU 尖峰 → 耗时波动
- 某算法模块内存占用高且 CPU 热点集中 → 优化该模块可同时降内存、降 CPU、降耗时

如果只看一个维度，很容易"头疼医头、脚疼医脚"。组合分析能定位到**多维度共振的根因模块**。

## 阶段 0：维度选择

**进入技能后，第一步必须询问用户选择优化维度。**
任何组合都支持——用户可能只想分析内存+脚本，那就不做 CPU 和耗时分析；可能只想看耗时波动，那就只做耗时分析。按需来，不强制全选。

使用 `AskUserQuestion` 工具，参数 `multiSelect: true`，**固定 4 个选项，全选 = 完整画像**：

```
问题：请选择要分析的优化维度（可多选，全选 = 完整画像）：
选项：
  1. "脚本代码防护" — .sol 脚本的异常防护与代码性能问题
  2. "内存分析" — C++ 非托管内存分配热点（需 PerfView + 进程运行）
  3. "CPU 热点分析" — CPU 调用热点函数/模块（需 VTune + 管理员权限）
  4. "耗时与波动分析" — 流程/模块耗时统计与波动归因（需 ModuleFrame 日志）
```

> **为什么是 4 个选项（无"全部维度"独立选项）？**
> `AskUserQuestion` 工具有硬性限制：每个问题最多 4 个选项。5 个选项（全部 + 4 个维度）会触发
> `InputValidationError: too_big: expected array to have <=4 items`。
> 因此将"全部维度"融入交互：用户勾选所有 4 个 = 完整画像，勾选部分 = 按需分析。

> **用户选择处理**：
> - 选了 4 个 → 全维度完整画像
> - 选了 1-3 个 → 只做选中的维度，不要催促或质疑
> - **一个都没选** → 提示"至少需要选择一个维度"，引导用户重新选择
> - 单一维度分析完成后，可以温和建议补充其他维度，但一次就够了，不要反复推荐

## 阶段 0.5：执行顺序告知与确认（⚠️ 多维度时必须执行）

**当用户选择了 ≥2 个维度时，必须先告知执行顺序并征求确认，再进入信息收集。**

### 为什么耗时分析排第一？

耗时分析只需 `ModuleFrame*.log` 日志文件，**完全不依赖 VisionMaster 进程**。
在所有维度中启动成本最低、执行最快，可以先行完成，为后续分析"抢时间"。

### 固定执行顺序与理由

| 顺序 | 维度 | 依赖 | 理由 |
|------|------|------|------|
| **第 1** | ⏱️ 耗时与波动 | 仅需日志文件，零进程依赖 | 启动最快，先行完成 |
| **第 2** | 🛡️ 脚本代码防护 | 仅需 .sol 文件 | 静态分析，不依赖进程 |
| **第 3** | 📊 内存分析 | 需进程运行 + ETW 采集 | 需要用户配合操作（加载方案/运行检测） |
| **第 4** | 🔥 CPU 热点分析 | 需进程运行 + 管理员权限 | 采集时间长，放在最后；如失败可提前终止 |

> **原则**：无依赖先跑（耗时、脚本），需要进程的后跑（内存、CPU）。

### 告知模板

```
根据你选择的 [N] 个维度，分析顺序如下：

  [如果选了耗时] [1] ⏱️ 耗时与波动 — 最先执行（只需日志文件，最快完成）
  [如果选了脚本] [2] 🛡️ 脚本代码防护 — 静态分析 .sol 文件
         ⚠️ 如发现问题且后续有内存/CPU 分析，默认「只记录不修改」
  [如果选了内存] [3] 📊 内存分析 — 需 VM 进程运行，分 3 阶段采集
  [如果选了 CPU] [4] 🔥 CPU 热点分析 — 需管理员权限

这个顺序遵循"无依赖先跑"原则。是否要调整？确认后开始收集信息。
```

> - 用户同意 → 进入阶段 1（信息收集）
> - 用户要求调整 → 按用户指定顺序执行
> - 用户只选了 1 个维度 → 跳过阶段 0.5，直接进入阶段 1

## 阶段 1.5：可用性检查

在开始执行前，**必须对用户所选维度做可用性预检**。这个预检分成两步：

### 第一步：组合技能做快速文件/进程检查

只检查不需要调用子技能就能判断的事项：

| 维度 | 检查项 | 检查方法 |
|------|--------|---------|
| 🛡️ 脚本 | .sol 文件存在 | `test -f "<sol路径>"` |
| 📊🔥 | 目标进程存在 | `tasklist \| grep -i "<进程名>"` |
| ⏱️ 耗时 | (1) 日志目录存在且含 ModuleFrame*.log | `test -d "<日志目录>" && ls "<日志目录>"/ModuleFrame*.log 2>/dev/null` |
| ⏱️ 耗时 | (2) 日志样本量是否充足 | 统计流程执行次数，< 100 次告警（详见下方「耗时日志样本量检查」） |

### 耗时日志样本量检查（必须在预检阶段完成）

**在调用 vm-execution-time-analyzer 之前**，先快速统计日志中的流程执行次数：

```bash
# ⚠️ 不能用 "流程.*模块" 匹配——实际日志没有这两个中文词。
# 正确匹配：流程执行行 = "status=free" + "executeCount="（log_parser.py 用此正则）
# 示例行: [CProcedure::ReportProcedureStatus@623] Report procedure status  id=10001  status=free  time=246.459  executeCount=1
grep -c "status=free.*executeCount=" "<日志目录>"/ModuleFrame*.log 2>/dev/null
```

根据结果决定是否告警：

| 执行次数 | 处理 |
|---------|------|
| ≥ 200 次 | ✅ 无告警，直接进入分析 |
| 100–199 次 | ⚠️ 告知用户"样本量中等，统计结论基本可信" |
| 20–99 次 | ⚠️ 告知用户"样本量偏小（N=XX），CV/VCI 极端值影响较大，建议多跑一些检测后重新分析" |
| < 20 次 | 🔴 告知用户"样本量严重不足（N=XX），统计结论不可靠。是否仍然分析？[继续/先跑检测]" |

> **告警示例**：
> "⚠️ 耗时日志中仅检测到 20 次流程执行记录。变异系数(CV)和波动贡献指数(VCI)在
> 小样本下容易被极端值影响，统计结论的置信度有限。建议先连续运行检测 ≥200 次
> 后再分析。是否仍然继续？"

> **为什么是 100 次？** 变异系数 (CV) 在样本量 < 30 时置信区间很宽，到 100 以上才
> 趋于稳定。VCI 依赖 CV × 流程占比，小样本下排名可能不准确。200 次是一个保守阈值。

> **即使用户选择继续**：正常执行分析，但在综合报告的耗时章节（5.1 方案耗时概况）中
> 显著标注样本量限制，如 `⚠️ 仅 20 次执行记录，CV/VCI 统计结论置信度有限`。

### 第一步-补充：输出路径 ASCII 安全预检

**在调用任何子技能之前，必须检查输出目录路径是否含非 ASCII 字符。**

```bash
# 检查输出路径是否纯 ASCII
echo "<输出目录>" | grep -qP '[^\x00-\x7F]' && echo "NONASCII" || echo "ASCII"
```

若含非 ASCII 字符：
- **自动替换为 ASCII 安全路径**：
  - 将 `CPU分析` → `CPU_Analysis`、`内存分析` → `Memory_Analysis`、`耗时分析` → `Time_Analysis`、`脚本分析` → `Script_Analysis`
  - 整个输出父目录也使用纯英文（如 `C:\VM_Optimization\<方案名>_<日期>\`）
- 告知用户路径已调整为纯英文路径以避免 PowerShell 脚本中文编码问题
- 同步更新所有子技能调用的输出路径参数

> **为什么必须纯 ASCII？** 实战验证：PowerShell 脚本（`vtune_collect.ps1`、`perfview_unmanaged.ps1`）
> 在 bash → PowerShell 调用链中，含中文的路径参数会被损坏为乱码（如 `CPU分析` → `CPU鍒嗘瀽`），
> 导致：①脚本创建的实际目录与 Agent 操作目录不一致；②`confirm.txt` 信号文件路径不匹配；
> ③采集等待超时（300s）。纯 ASCII 路径可完全规避此类问题。

### 第二步：子技能自己检查工具

**不要在组合技能里写死工具安装指引。** 每个子技能内部已经有完善的工具检测逻辑和安装引导（vtune-analyzer 有 VTune 未安装时的 A/B 选项、perfview-analyzer 有 PerfView/xperf 未找到时的下载链接和询问模板）。组合技能的工作是：**正常调用子技能，让子技能自己去发现和提示**。

```
做法：直接按执行计划调用子技能。

如果子技能在内部检测到工具缺失，它会自动：
  - vtune-analyzer → 展示选项 A（安装 VTune）+ 选项 B（WPR 替代），让用户选择
  - perfview-analyzer → 询问 PerfView.exe 路径，或提供下载链接
  - vm-script-protection → 提示 .NET Framework 版本不足，给出下载地址
  - vm-execution-time-analyzer → 主要是 Python 和日志文件，常规环境都有
```

### 检查结果处理

| 场景 | 处理方式 |
|------|---------|
| 文件/进程检查通过 | 直接进入阶段 2，按计划调用子技能 |
| .sol 文件不存在 | 请用户确认路径，跳过脚本维度 |
| 进程不在运行 | 请用户先启动进程并进入目标场景，在此期间可以先执行不依赖进程的维度（耗时、脚本） |
| 日志目录为空 | 请用户确认路径或先运行方案产生日志，跳过耗时维度 |

**核心原则**：
- 组合技能只做文件/进程的快速预检
- 工具安装引导由各子技能自己处理，不要在本技能中重复维护
- 子技能在工具缺失时自然会暂停并询问用户——这正是期望的行为
- 如果子技能因工具缺失而中断（用户选择跳过），记录原因，继续下一个维度

## 阶段 2：执行编排

### 执行原则

1. **先告知执行顺序**：如用户选了 ≥2 个维度，必须先在阶段 0.5 展示执行顺序并征求确认
2. **先告知再执行**：每个维度开始前，口头总结已收集的参数
3. **无依赖先跑**：耗时（日志文件）最先，脚本（.sol 文件）其次，需要进程的内存/CPU 最后
4. **每次只调一个技能**：技能调用会接管对话，必须等上一个完成再调下一个
5. **传递上下文**：调用子技能时，直接把已收集的参数传递给子技能，跳过其重复询问
6. **记录输出**：每个子技能完成后，立即记录其核心输出（报告路径、关键发现、TOP 热点列表）
7. **验证独立报告**：每个维度完成后，必须确认子技能生成了独立的分析报告（.md / .html），
   若未生成，在进入下一维度之前补生成。综合报告引用这些独立报告，而不是替代它们。
8. **故障隔离**：任一维度失败不影响其他维度执行，也不阻断综合报告生成。见下方「故障隔离机制」。

### 推荐执行顺序

**根据用户选中的维度，只执行对应的 Step，跳过来选中的。**

```
确认计划 → 用户确认
  ↓
Step A: [如果选了耗时] 调用 vm-execution-time-analyzer
  必出报告：<输出目录>/Time_Analysis/耗时分析报告.html
  记录：流程数、模块数、TOP VCI 模块、TOP 耗时流程
  ↓
Step B: [如果选了脚本] 调用 vm-script-protection
  必出报告：<sol目录>/<方案名>_代码防护报告.md
  记录：ShellModule 数、风险数、防护类型分布
  ↓
  ⚠️ 决策分支（见下方「Step B 特殊流程」）
  ↓
Step C: [如果选了内存] 调用 perfview-analyzer
  必出报告：<输出目录>/Memory_Analysis/PerfView_分析报告_<进程名>_<日期>.md
  记录：三阶段内存对比、TOP 业务模块排名、关键调用栈
  ↓
Step D: [如果选了 CPU] 调用 vtune-analyzer
  必出报告：<输出目录>/CPU_Analysis/raw_report/VTune_Analysis_Report.md
  记录：TOP 热点函数、TOP 模块 CPU 占比
  ↓
阶段 3: 生成综合报告（引用以上所有独立报告）
```

示例：用户选了脚本+内存 → 执行 Step B → 决策分支 → Step C → 阶段 3
示例：用户只选了耗时 → 执行 Step A → 阶段 3

> **Step C/D 执行前提**：确保用户已将进程置于目标场景状态（如已加载方案、准备运行检测）。
> 如果进程不在运行中，先让用户启动并进入目标状态。

### Step B 特殊流程：脚本维度与后续分析的联动决策

**为什么脚本分析的修改决策需要特殊处理？**

脚本中发现的代码问题（如循环内大对象分配、重复 LINQ）可能是下游内存/CPU 问题的**根因**。
如果在脚本分析阶段就修改了 `.sol` 文件，后续内存/CPU 分析将在**修改后的方案**上运行——
此时原方案的根因已被"修复"，内存热点和 CPU 热点数据就无法反映真实问题。

> **核心原则**：先诊断、后治疗。在完整画像出炉前，保留"案发现场"。

#### 决策树

脚本分析（vm-script-protection）完成并输出分析报告后，按以下逻辑处理：

```
脚本分析是否发现了需要修改的问题？
  │
  ├── 否（0 处自动注入项，仅警告或全部已防护）
  │     → 继续执行后续维度（如有），无额外决策
  │
  └── 是（存在 try-catch/除零/越界/死循环等自动注入项，或用户关心的性能问题）
        │
        ├── 用户未选内存+CPU（仅脚本单维度）
        │     → 按 vm-script-protection 的正常确认流程处理
        │     → 用户确认 → 修改 .sol → 完成
        │     → 用户跳过 → 记录问题但不修改
        │
        └── 用户选了内存 或 CPU（存在后续分析）
              → 默认："不修改，仅记录问题详情"
```

#### 默认选项：不修改，仅记录

当脚本发现问题且后续还有内存/CPU 分析时，向用户展示：

```
🛡️ 脚本分析发现 [N] 处需要修改的问题。

⚠️ 由于你选择了后续的内存/CPU 分析，默认建议「暂不修改脚本」——
先在原始方案上完成全维度分析，确认根因后再统一修改。
修改脚本后再分析内存/CPU，会掩盖"脚本问题 → 内存/CPU 热点"的因果链。

  [A] 不修改，仅记录问题详情（推荐）— 综合报告中将保留完整的问题描述和前后代码对比
  [B] 修改脚本，但保存为备份方案 — 按备份名保存修改后方案，原方案不变，用于后续分析
  [C] 修改原方案 — 直接修改 .sol（⚠️ 后续内存/CPU 分析将在修改后的方案上运行）
```

> **选项 B 的实现**：
> ```bash
> # 修改前先备份：将 vm-script-protection 的输出保存到 <方案名>_脚本优化.sol
> cp "<sol路径>" "<sol目录>/<方案名>_脚本优化.sol"
> # 然后对备份方案执行修改，原方案保持不变
> # 后续内存/CPU 分析使用的仍是原方案
> ```

> **选项 C 的警告**：如果用户坚持修改原方案，必须明确告知后果——
> "后续内存/CPU 分析将使用修改后的方案，如果内存/CPU 问题是由脚本代码引起的，
> 这些问题的根因将无法被发现。综合报告的交叉关联分析中会标注此限制。"

#### 记录要求

无论用户选择哪个选项，脚本分析中发现的**每一个问题都必须详细记录**，包括：

- 问题类型、严重等级
- 出问题的模块名（displayName + fullPath）
- 原始代码（before）
- 修改方案（after）
- 位置（行号）

这些记录在综合报告的「脚本代码质量分析」章节完整呈现，即使未实际修改。

#### Step B 报告验证（必须执行）

**无论用户选择 A/B/C 哪个选项，vm-script-protection 的 Step 12（报告生成）必须执行。**
综合技能在 Step B 完成后应立即验证：

```bash
# 验证脚本分析报告是否已生成
test -f "<sol目录>/<方案名>_代码防护报告.md" && echo "✅ 脚本报告已生成" || echo "❌ 缺失——立即补生成"
```

若报告缺失：
- 利用 Step 4-5 已收集的分析数据（问题类型、位置、before/after 代码对比）
- 按 vm-script-protection 的 Step 12 模板格式补生成 `.md` 报告
- 报告内容中所有问题标注为 `[未应用]`（因综合技能策略为「先诊断、后治疗」）

**脚本报告同步到统一输出目录**：

无论 vm-script-protection 将报告生成在何处，综合技能必须将脚本报告**同步复制**到统一输出目录：

```bash
# 1. 确认源报告存在
test -f "<sol目录>/<方案名>_代码防护报告.md" || echo "MISSING"

# 2. 复制到统一输出目录（与其他三维度报告并列）
mkdir -p "<输出目录>/Script_Analysis"
cp "<sol目录>/<方案名>_代码防护报告.md" "<输出目录>/Script_Analysis/<方案名>_代码防护报告.md"
```

> **为什么需要复制？** 统一输出目录下已有 CPU_Analysis/、Memory_Analysis/、Time_Analysis/ 三个子目录
> 及对应报告，脚本报告不应缺失。将报告集中存放便于后续查阅和综合报告的附录 A 引用。

> **为什么必须验证？** vm-script-protection 的原始流程中 Step 12 依赖用户确认（Step 5d），
> 但综合技能可能拦截了用户交互（如默认"仅分析不修改"），导致 Skill 调用提前结束。
> 无论子技能是否执行 Step 12，综合技能都必须确保独立报告存在——这是综合报告的输入源。

### 调用子技能的方式

使用 `Skill` 工具调用子技能。关键技巧：**调用前，先向用户口头总结已收集的参数，
然后调用 Skill 工具。因为对话上下文已包含所有信息，子技能加载后会看到这些上下文，
自然跳过已确认的询问。**

具体做法：
1. 调用前，对用户说："现在开始 [维度] 分析，之前已确认参数：.sol 路径 = X，防护范围 = 全部。"
2. 然后调用 `Skill: vm-script-protection`
3. 子技能加载后，看到上下文中已有参数，会自动跳过重复询问

如果子技能仍然询问已知参数（极少数情况），直接替用户回答即可，不要让用户再输入一遍。

### ⚠️ Step C 特殊流程：内存三阶段采集的强引导交互

内存分析（perfview-analyzer）需要用户在采集期间配合切换方案状态。**这是整个分析流程中交互最密集的环节，必须按以下固定流程执行，不得跳过任何一步：**

#### C.0 — 采集前告知（在调用 perfview-analyzer 之前）

**第一步：显式询问用户是否调整采集时长（⚠️ 不可跳过）**

```
内存分析默认分三个阶段采集：

  阶段 1「空方案」：30s（进程空闲基线）
  阶段 2「加载方案」：30s（加载 .sol 方案过程中，方案较大可设 60-120s）
  阶段 3「运行检测」：60s（执行实际工作负载，可设 60-180s）

是否需要调整任何阶段的采集时长？直接回复确认使用默认值，或指定调整。
```

> **为什么必须问？** 不同方案的加载时间差异巨大（简单的几秒，复杂的几十秒）。
> 用户最清楚自己的方案需要多长时间加载和运行。默认值只是一个合理起点。

**第二步：用户确认时长后，告知完整的三阶段方案**

```
内存分析采用三阶段对比采集，每个阶段按以下固定流程：

  阶段 1「空方案」({duration1}s)：先启动 PerfView 采集 → 立即提示你"保持空方案状态不动"
  阶段 2「加载方案」({duration2}s)：先启动 PerfView 采集 → 立即提示你"加载 {方案名}.sol"
  阶段 3「运行检测」({duration3}s)：先启动 PerfView 采集 → 立即提示你"连续运行检测"

⚠️ 关键：PerfView ETW 一旦启动就开始记录，所以必须"先启动采集、再让你操作"。
每个阶段我会先确认，再执行。准备好了吗？
```

#### C.0.5 — 采集前技术前置（⚠️ 必须在 Step C 首次调用 PerfView 前执行）

**以下三项是实战中验证的"不执行就卡死"的硬性前提：**

**1. 设置符号路径（防止 Empty Symbol Path 弹窗阻塞）**

```bash
# 必须先设置 _NT_SYMBOL_PATH！否则 PerfView 在 stop/merge 阶段弹出
# "Empty Symbol Path" 对话框等待用户点击，导致整个流程卡死。
# 源码验证：App.cs:1084 UserOKWithSymbolServerGui() → EmptySymbolPathDialog
# ⚠️ 只用本地缓存，不追加微软公网符号服务器（perfview-analyzer 原则：触网须用户同意）
export _NT_SYMBOL_PATH="C:\Symbols"
```

**2. 必须用 `/LogFile=` 而非 `/NoView`（抑制 GUI）**

```bash
# /NoView 只跳过结果查看，GUI 仍然创建（会触发 Empty Symbol Path 弹窗）
# /LogFile=collect.log 彻底抑制 GUI，是 PerfView 官方脚本模式的正确用法
# 源码验证：UsersGuide.htm:7623 "Using PerfView in Scripts (/LogFile qualifier)"
#          CommandLineArgs.cs:578 "For batch scripts/automation use /LogFile instead"
MSYS_NO_PATHCONV=1 "E:/PerfView/PerfView.exe" \
  "/DataFile=C:/VM_Optimization/.../VisionMaster_Phase.etl" \
  /MaxCollectSec:30 /CircularMB=10000 /BufferSizeMB=4096 \
  /KernelEvents=Default,VirtualAlloc,VAMap /CpuSampleMSec:1 \
  /LogFile=collect.log /AcceptEula collect
```

**3. 必须用 `MSYS_NO_PATHCONV=1`（防止 Git Bash 路径转换）**

```
# 不加此前缀：/CircularMB=10000 → C:/Program Files/Git/CircularMB=10000（命令失败）
# 加上此后：/CircularMB=10000 → /CircularMB=10000（正确传递）
# 所有 PerfView 命令、所有 xperf 命令均需此前缀
```

**4. 采集完成后用文件检查代替后台任务等待**

```bash
# 不要依赖 Bash 后台任务追踪器（PerfView 进程退出后追踪器可能不感知）
# 直接检查 .etl.zip 文件是否存在且大小 > 0：
ls -lh "<输出目录>/Memory_Analysis/"*.etl.zip
# 文件存在且大小合理（通常 30-80 MB）即表示采集成功

**5. 输出路径 ASCII 化检查（必须在所有采集命令执行前确认）**

```bash
# ⚠️ 确保 PerfView/xperf/VTune 输出路径不含中文。
# 中文路径在 PowerShell 脚本内部会被损坏为乱码（如 CPU分析 → CPU鍒嗘瀽），
# 导致信号文件路径不匹配、采集等待超时（300s）。
# Phase 1.5 已做预检，C.0.5 再次确认。
echo "<输出目录>" | grep -qP '[^\x00-\x7F]' && echo "⚠️ 路径含非 ASCII！" || echo "✅ 路径安全"
```
```

#### C.1 — 逐阶段执行（在 perfview-analyzer 技能内）

perfview-analyzer 接管后，组合技能无需额外干预。但需确保以下节奏：

**每个阶段的固定节奏：**

```
1. 口头确认："开始阶段 N「XXX」，采集 X 秒，准备好了吗？"
2. 用户确认后 → 立即启动 PerfView collect 命令（C.0.5 中的模板）
3. 命令启动的同一时刻 → 立即用中文提示用户执行操作：
   - 阶段 1："🔴 采集已启动（30s）！请保持空方案状态，不做任何操作。"
   - 阶段 2："🔴 采集已启动（30s）！请立即加载 {方案名}.sol 方案。"
   - 阶段 3："🔴 采集已启动（60s）！请立即连续运行检测。"
4. 等待 MaxCollectSec+30s 后，检查 .etl.zip 文件是否存在
5. 确认文件生成 → 告知用户"✅ 阶段 N 采集完成（XX MB）" → 进入下一阶段
```

> **⚠️ Bash 工具 description 必须用中文**：Bash 命令的 `description` 参数会显示给用户。
> 正确：`"阶段2：加载方案内存采集（30s）"` | 错误：`"Phase 2: Load solution memory collection"`
> 用户看到英文 "Stop PerfView collection Phase 1" 无法区分"卡住了"和"阶段1采集结束"。

> **为什么用户看到的"启动采集"到"采集已启动"间隔了几分钟？**
> 原因是 Bash 的 "Command running in background" 消息在 PerfView 启动之前就打印了，
> 而 PerfView 启动有 3-5 秒的初始化延迟（加载 ETW providers）。再加上 collect 内部
> WaitForStart 可能等待若干秒。修正：**用户确认后立即显示"采集已启动！请立即…"**，
> 不要再等待命令的实际返回值。ETW 从 start 命令执行瞬间就开始记录。

#### C.2 — 异常处理

| 场景 | 处理 |
|------|------|
| 用户操作来不及（如方案加载超过 30s） | 告知用户，该阶段重新采集，时长翻倍 |
| 某阶段采集失败（.etl.zip 未生成或大小为 0） | 告知用户具体错误，检查 LogFile 输出，询问是否重试或跳过 |
| 进程中途退出 | 确认用户是否手动关闭，重新启动后从失败阶段继续 |
| **PerfView 弹出 Empty Symbol Path 对话框** | **C.0.5 已设置 `_NT_SYMBOL_PATH` 来预防。** 若仍出现：`export _NT_SYMBOL_PATH="C:\Symbols"` 后重试 |
| **PerfView 命令报 "Extra positional parameter"** | **C.0.5 已用 `MSYS_NO_PATHCONV=1` 预防。** 若仍出现：检查所有 `/Xxx=yyy` 参数前是否有空格被 Git Bash 截断 |

## 阶段 3：综合报告生成

所有子技能执行完毕后，基于收集到的输出，生成一份**结构化综合报告**。

### 报告输出位置

所有报告统一放在阶段 1 收集的输出目录下，目录结构如下：

```
<输出目录>/                          ← 用户在阶段 1 指定的目录
├── 综合优化报告_<方案名>_<日期>.md    ← 综合报告（本技能产出）
├── Script_Analysis/                  ← vm-script-protection 输出
├── Memory_Analysis/                  ← perfview-analyzer 输出（.etl.zip + 报告）
├── CPU_Analysis/                     ← vtune-analyzer 输出（CSV + 报告）
└── Time_Analysis/                    ← vm-execution-time-analyzer 输出（HTML 报告）
```

**实现方式**：
- 综合报告使用 `Write` 工具保存到 `<输出目录>/综合优化报告_<方案名>_<日期>.md`
- 子技能输出路径控制能力各不相同，调用时尽量引导到输出目录的子目录下：
  | 子技能 | 输出路径控制 | 做法 |
  |--------|------------|------|
  | vm-script-protection | 不支持自定义输出，修改后的 .sol 和 .bak 放在原始 .sol 同目录 | 将原始 .sol 路径记录下来，附录 A 如实标注 |
  | perfview-analyzer | 采集脚本输出 ETL 到当前工作目录，报告由 LLM 生成 | 调用前 `cd` 到 `<输出目录>/Memory_Analysis/`（需先 mkdir），使 ETL 文件落在该目录 |
  | vtune-analyzer | `-OutputDir` 参数控制原始结果目录，报告自动输出到 `<OutputDir>_report` | 传递 `-OutputDir "<输出目录>/CPU_Analysis/raw"`，则报告在 `<输出目录>/CPU_Analysis/raw_report/` |
  | vm-execution-time-analyzer | `--output` 参数控制 HTML 报告路径 | 传递 `--output "<输出目录>/Time_Analysis/耗时分析报告.html"` |
- 调用前先 `mkdir -p "<输出目录>/<子目录>"` 确保目录存在
- 如果子技能的实际输出路径与预期不同，在附录 A 中如实记录

### 报告生成方法

1. **保留原始报告**：每个子技能会生成自己的完整报告（.md / .html / CSV），这些是独立可读的。
   综合报告**不替代**它们，而是在它们之上做提炼+关联+排序。
2. **维度章节要详实**：从子技能输出中提取充足的细节（数值、排名、模块名、函数名），
   确保读者只看综合报告也能掌握该维度的核心发现，不必翻原始报告。
3. **按模板生成选中维度章节**：只生成用户选中维度的章节，未选中的维度不出现在报告中。
4. **交叉关联分析是核心增量**：将各维度的热点模块对齐到统一模块名，构建关联矩阵。
5. **优化建议按 P0→P3 分级**：每条建议标注来源维度，多维度共振问题自动获得更高优先级。

### 故障隔离机制：某维度卡死/失败时如何兜底

**即使部分维度失败，已完成维度的独立报告和综合报告仍必须生成。**

#### 故障分类

| 故障类型 | 特征 | 处理方式 |
|---------|------|---------|
| **子技能内阻塞** | 采集/等待命令无响应，用户催促"卡死了" | 用户主动终止（Ctrl+C 或发送"跳过"），标记该维度为"用户终止"，继续下一维度 |
| **子技能执行错误** | 命令返回非零 exit code 或明确错误信息 | 记录错误原因，标记为"执行失败"，继续下一维度 |
| **工具缺失** | PerfView/VTune 未安装，子技能提示后用户选择跳过 | 标记为"工具缺失"，继续下一维度 |
| **进程不在运行** | 内存/CPU 分析前进程已退出，用户无法/不愿重启 | 标记为"进程不可用"，跳过该维度 |

#### 状态追踪

## 综合报告模板

**核心原则**：
1. **只输出用户选中维度的章节**。比如用户只选了脚本+内存，报告就只有 一（总览）、二（脚本）、三（内存）、六（交叉关联）、七（优化建议）、八（进一步排查）。未选中的维度章节直接跳过，不要留空标题或写"未分析"占位。
2. **每个维度章节要详实**——把子技能报告中的关键数据完整搬过来（热点排名、数值、模块名、函数名、风险等级），让读者不翻原始报告也能获得这个维度的全部核心信息。不是写摘要，是搬运+整合。
3. **交叉关联和优化建议是综合报告独有的增量价值**，这部分子技能报告里没有，要重点投入。
4. **各维度小结的写作要求**：每个维度的"本维度小结"不应只是"发现 3 个问题，建议优化"这种空洞的话。要写成**一段完整的诊断意见**，包含：① 该维度最突出的问题是什么（引用具体数据）② 这个问题对方案的影响有多大（稳定性/性能/资源）③ 与哪些其他维度可能有关联（为第六章的交叉关联埋下伏笔）。长度控制在 3-5 句，但要言之有物。

**重要**：下文中 `[ ]` 包裹的内容是需要从子技能输出中提取填充的位置，不要原样输出。

## 六、交叉关联分析

> 这是本报告的核心增量——将各维度的热点对齐到统一的模块视角。
> - **2 个及以上维度**：核心工作是构建关联矩阵 + 找出多维度共振的"重灾区"。
> - **仅 1 个维度**：跳过 6.1（关联矩阵）和 6.2（共振热点）和 6.3（因果链分析），只保留 6.4（独立问题清单），将标题改为"六、问题汇总与优先级排序"。

### 6.1 多维度关联矩阵

**矩阵的列数取决于选中维度数。** 固定列：模块名 + 命中维度 + 综合风险（3 列），每选中一个维度增加该维度的风险/热点列。比如选了脚本+内存（2 个维度），矩阵共 5 列：模块名、脚本风险、内存热点、命中维度、综合风险。

| 模块名 | 🛡️脚本风险 | 📊内存热点 | 🔥CPU热点 | ⏱️耗时/VCI | 命中维度 | 综合风险 |
|--------|-----------|-----------|----------|------------|---------|---------|
| [模块A] | 🔴 除零+越界 | TOP 2 (128MB) | TOP 3 (12%) | VCI TOP 1 | 4/4 | 🔴 极高 |
| [模块B] | 🟢 无 | TOP 1 (256MB) | TOP 1 (25%) | VCI TOP 3 | 3/4 | 🔴 高 |
| [模块C] | 🟡 循环拼接 | - | - | VCI TOP 5 | 2/4 | 🟡 中 |
| ... | ... | ... | ... | ... | ... | ... |

**矩阵构建方法（按步骤执行）**：

1. **收集候选**：从每个维度的分析结果中提取热点/风险模块名，形成初始列表。取各维度的 TOP 5-10，加上脚本分析中发现的所有有风险模块。
2. **名称对齐**：按照 [注意事项 §4](#注意事项) 的方法，将 DLL 名/类名统一为中文业务名称。对于无法确定对应关系的，保留原始名称并标注 `[待确认]`。
3. **逐模块填表**：对每个候选模块，检查它在各维度中是否出现——
   - 脚本维度：是否有防护缺失或性能问题？风险等级？
   - 内存维度：是否在热点 TOP N 中？排名和分配量？
   - CPU 维度：是否在热点 TOP N 中？排名和占比？
   - 耗时维度：是否在 VCI TOP N 中？排名和评级？
   如果某维度未分析，该列留空即可。
4. **计算综合风险**：
   - 🔴 极高：≥3 个维度命中，且至少一个维度是 TOP 1-2
   - 🔴 高：2+ 个维度命中，或多个单维度 TOP 3
   - 🟡 中：1-2 个维度命中，排名在 TOP 5 之后
   - 🟢 低：仅单维度低风险
5. **排序输出**：按综合风险从高到低排列，同一等级内按命中维度数降序

## 注意事项

1. **不要并行调用子技能**：每个子技能都需要与用户交互，同时调用会导致混乱。严格顺序执行。
2. **传递上下文**：每次调用子技能前，口头总结已知参数再调用，让对话上下文自然传递。
3. **记录热点模块名**：每个维度分析完成后，立即记录 TOP 热点模块的名称列表及原始命名形式（DLL 名/类名/中文名），用于阶段 3 的交叉关联分析。
4. **模块名对齐**（交叉关联分析的关键前提）：四个子技能输出的模块名形式不同——
   - 脚本分析用 ShellModule 的 `displayName`（中文，如"高精度匹配"）
   - 内存分析用 DLL 名（如 `MVDAlgorithm.dll`），perfview-analyzer 有 `references/VisionMaster模块映射表.md`
   - CPU 分析用 DLL/EXE 名，vtune-analyzer 输出中可能包含函数名
   - 耗时分析用模块类名（如 `CMatchModule`），vm-execution-time-analyzer 有 `references/module_names.json`
   
   **对齐方法**：
   - 优先使用各子技能自带的映射文件（perfview 的模块映射表、耗时分析的 module_names.json）
   - 如果自带映射不完整，结合 DLL 名、类名、函数名的关键词推断对应关系（如 DLL 中含 `Match` → 耗时分析中 `CMatchModule` → 脚本中"高精度匹配"）
   - 在交叉关联矩阵中统一使用**中文业务名称**，括号内附原始名称
   - 如果实在无法确定对应关系，标注 `[待确认]`，不要强行关联
5. **子技能失败处理**：

   分为两类：**预检失败**（组合技能在阶段 1.5 发现，不调用子技能）和 **子技能内失败**（调用子技能后由子技能自行处理或执行中出错）。

   **预检失败**（阶段 1.5 第一步发现，不调用子技能）：
   | 场景 | 处理方式 |
   |------|---------|
   | .sol 文件不存在 | 请用户确认路径，用户修正后继续；用户无法提供则跳过脚本维度 |
   | 日志目录为空 | 请用户确认路径或先运行方案产生日志，用户无法提供则跳过耗时维度 |
   | 进程不在运行 | 请用户先启动进程并进入目标场景。在此期间可以先执行不依赖进程的维度（脚本、耗时）。用户无法启动则跳过内存+CPU 维度 |

   **子技能内失败**（调用子技能后发生）：
   | 场景 | 处理方式 |
   |------|---------|
   | 工具缺失（VTune/PerfView/xperf 等） | **不要组合技能自己提供下载链接。** 子技能内部已有完善的检测逻辑和安装引导——调用子技能，让它自己向用户展示选项（安装/跳过/替代），用户自行决定。组合技能只记录最终结果 |
   | 子技能执行中出错（非工具缺失） | 记录错误信息，该维度在报告中标注"分析失败"，在进一步排查建议中给出替代方案 |
   | 用户主动选择跳过某维度 | 尊重用户决定，记录原因，继续下一个维度 |

   **核心原则**：
   - 一个维度失败不影响其他维度继续执行
   - 工具安装引导由子技能处理，组合技能不越俎代庖
   - 预检失败 → 组合技能处理；工具缺失 → 子技能处理；执行错误 → 组合技能记录
6. **全程使用中文**：所有交互、提示、报告均使用简体中文。
